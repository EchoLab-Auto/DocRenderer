# 文档组织规则

本文档定义 ProDoc 项目自身的文档体系结构、编写规范和维护准则。所有向本项目贡献文档的参与者都必须遵循以下规则。

## 一、目录结构

项目文档统一存放在 `docs/` 目录下，按以下四层级组织：

```
docs/
├── index.md                    # 文档系统首页
├── before-develop/             # 开发前必读
│   ├── index.md
│   └── doc-organization.md     # 本文档
├── guide/                      # 使用指南
│   ├── index.md
│   ├── getting-started.md
│   ├── configuration.md
│   └── prodoc-format.md
└── reference/                  # API 参考
    ├── index.md
    └── ui-layout.md
```

### 1.1 各目录职责

| 目录 | 职责 | 目标读者 |
|------|------|----------|
| `docs/index.md` | 文档系统总入口，提供全局导航和项目概览 | 所有用户 |
| `docs/before-develop/` | 开发前必须阅读的设计原则、约束条件、风格规范 | 开发者、贡献者 |
| `docs/guide/` | 面向终端用户的使用说明，包括安装、配置、格式规范 | 框架使用者 |
| `docs/reference/` | 精确的 API 文档、布局参数、类型定义等速查资料 | 集成开发者 |

**原则：** 同一个主题的内容不要分散到多个目录。如果一篇文档同时涉及「使用方式」和「API 细节」，以文档的主要目的决定归属。

## 二、文件命名规范

### 2.1 基本规则

- 全部使用**小写字母**，单词间用**连字符 `-`** 分隔
- 必须使用 `.md` 扩展名
- 文件名应能概括文档内容，避免使用 `doc1.md`、`temp.md` 等无意义名称

**正确示例：** `doc-organization.md`、`getting-started.md`、`ui-layout.md`  
**错误示例：** `DocOrganization.md`、`getting_started.md`、`doc1.md`

### 2.2 索引文件

每个子目录必须包含一个 `index.md`，作为该章节的入口页面，承担以下职责：

1. 说明本章节的内容范围和目标读者
2. 列出本章节下的所有子文档（带链接）
3. 提供向后一页的导航指引

### 2.3 根目录文件

`docs/index.md` 是文档系统的首页，必须包含：

1. 项目一句话简介
2. 核心特性列表
3. 快速开始命令
4. 章节导航链接

## 三、Frontmatter 规范

所有文档文件必须在开头包含 YAML frontmatter，至少包含以下两个字段：

```yaml
---
title: "文档标题"
order: 1
---
```

### 3.1 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `title` | `string` | 是 | 文档标题，用于导航树和页面显示 |
| `order` | `number` | 是 | 排序权重，同级节点按此值升序排列 |

### 3.2 排序规则

- 同级节点按 `order` 值**升序**排列，值越小越靠前
- `order` 建议以 1、2、3... 连续编号，而不是 10、20、30（除非预期中间会插入新文档）
- 每个目录的 `index.md` 通常设 `order: 1`，确保排在最前

## 四、文档内容规范

### 4.1 标题层级

- 文档中的**第一个标题（H1）** 必须与 frontmatter 中的 `title` 保持一致
- 标题层级必须连续，不允许跳级（如 H1 后直接 H3）
- 同一文档内 H1 只能出现一次

### 4.2 内部链接

文档之间的相互引用必须使用**相对路径**：

```markdown
<!-- 正确 -->
[配置指南](./configuration.md)
[UI 布局定义](../reference/ui-layout.md)

<!-- 错误 -->
[配置指南](https://github.com/.../configuration.md)
[配置指南](/docs/guide/configuration.md)
```

### 4.3 代码块

- 展示配置文件或代码示例时，必须标注语言标识符
- 涉及文件路径的代码块，应注释说明该文件在仓库中的位置

```markdown
```ts
// packages/prodoc-core/src/types.ts
interface ProDocNode {
  id: string;
  title: string;
}
```
```

### 4.4 表格

- 表格必须包含表头
- 同一表格内对齐方式保持一致
- 复杂表格超过 6 列时，考虑拆分为多个表格或改用列表

## 五、新增文档流程

当需要新增一篇文档时，按以下步骤操作：

1. **确定归属目录**：根据内容性质判断应放入 `before-develop/`、`guide/` 还是 `reference/`
2. **命名文件**：按照 2.1 节的命名规范确定文件名
3. **分配 `order`**：查看同目录下现有文档的 `order` 值，选择一个合适的位置
4. **更新父索引**：在所属目录的 `index.md` 中添加新文档的链接
5. **更新根索引**（可选）：如果文档重要性较高，在 `docs/index.md` 的章节导航中提及

## 六、维护要求

- **链接有效性**：修改文件名或目录结构后，必须检查并更新所有相关的内部链接
- **索引同步**：新增、删除或重命名文档后，必须同步更新对应目录的 `index.md`
- **版本一致性**：文档中涉及的版本号、路径、命令等必须与实际代码保持一致
- **定期审查**：建议在每个版本发布前审查一次文档，删除过期内容

## 七、与其他文档的关系

| 文档 | 关系说明 |
|------|----------|
| `prodoc-format.md` | 定义的是 **ProDoc 框架** 所支持的文档格式规范（面向框架使用者）；本文档定义的是 **本项目** 的文档组织规则（面向项目贡献者） |
| `README.md`（根目录） | 面向 GitHub 访客和初次接触者，内容偏项目概览和快速上手；`docs/` 下的文档面向需要深入了解的使用者和开发者，内容更详尽 |

## 八、示例：新增一篇文档的完整操作

假设需要在 `guide/` 下新增一篇「主题定制」文档：

**1. 创建文件 `docs/guide/theme-customization.md`：**

```markdown
---
title: "主题定制"
order: 4
---

# 主题定制

本文档介绍如何通过 CSS 变量自定义 ProDoc 的外观...

## 覆盖 CSS 变量

...
```

**2. 更新 `docs/guide/index.md`：**

```markdown
## 内容

- [快速开始](./getting-started.md)
- [配置指南](./configuration.md)
- [ProDoc 文档格式规范](./prodoc-format.md)
- [主题定制](./theme-customization.md)  ← 新增
```

**3. 检查 `docs/index.md`：** 确认是否需要更新首页导航。
